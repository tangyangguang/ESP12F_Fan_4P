# AGENTS.md

本文件记录本项目后续开发、调试和代码审查时必须遵守的原则性约定。所有协作者和自动化代理在修改本仓库前应先阅读并遵守。

## 项目定位

- 本项目是 `ESP12F_Fan_4P`，目标硬件为 ESP8266 ESP-12F / ESP-12E，用于四线 PWM 风扇控制。
- 设备主要通过局域网 Web 页面、本地按键和红外遥控操作。
- 默认访问地址为 `http://esp-fan.local/`，主要页面包括 `/fan`、`/config`、`/logs`。

## 最高工程原则

- 不管是新需求、bug 修复还是优化，都坚决不要背历史包袱。
- 不做打补丁式修复，不做临时方案，不为了少改代码而牺牲设计质量。
- 即使影响较多代码、模块或既有方案，也应选择最优方案和最佳实践。
- 如果最佳方案需要先调整架构、基础库或接口边界，应明确指出并推动正确方向，而不是在本项目内绕路。

## Esp8266Base 依赖边界

- 本项目依赖 Esp8266Base，仓库地址为 `https://github.com/tangyangguang/Esp8266Base`。
- 本项目内不得修改 Esp8266Base 库的任何文件，即使本地开发时通过 symlink 或本地路径引用该库。
- 如果发现问题属于 Esp8266Base 的 bug、设计缺陷或通用能力缺失，不要在本项目里为基础库打补丁。
- 遇到 Esp8266Base 相关问题时，应给出清晰的改进提示词或需求说明，由用户在 Esp8266Base 项目中处理。
- Esp8266Base 更新后，应优先阅读其 `CHANGELOG.md`，再阅读 README、API、Observability 文档和相关源码，最后判断本项目是否需要适配。
- 文档中描述 Esp8266Base 依赖时使用仓库 URL，不写个人电脑本地路径。

## 日志与配置审计

- 本项目直接使用 Esp8266Base 的文件日志能力，不再自行维护日志存储页面或日志文件轮转逻辑。
- `/logs` 页面使用 Esp8266Base 内置日志页面。
- 文件日志应记录启动分割、启动次数、用户操作、配置保存、WARN 及以上级别事件。
- NTP 同步后，日志应优先使用正确的日期时间。
- 配置写审计保持开启，且本项目不要求脱敏，便于个人调试观察。
- 项目默认保持 `-DESP8266BASE_LOG_LEVEL=0`，方便现场调试。
- 配置读审计由 Esp8266Base 的 `ESP8266BASE_CFG_READ_AUDIT_LEVEL` 单独控制；需要临时观察所有配置读取时，再显式加入：

```ini
-DESP8266BASE_LOG_LEVEL=0
-DESP8266BASE_CFG_READ_AUDIT_LEVEL=0
```

## 配置保存原则

- `/config` 页面保存参数时，只能对实际变化的字段执行写操作。
- 未变化字段不应触发 `setInt`、`setBool`、`setStr`，也不应产生 `no_change/skipped` 写审计日志。
- 普通配置保存不得遍历并保存红外码。
- 红外码只应在红外学习成功后保存。
- 多个配置项同时变化时，当前 Esp8266Base 会逐个 key 原子写入，因此保存较慢是可接受现象。
- 当前逐 key 写入方案下，单个 key 不应写坏；断电时可能出现部分 key 已保存、部分 key 保持旧值，但不应出现配置文件错乱。
- 如需优化多参数保存速度，应优先在 Esp8266Base 中设计事务型或批量配置提交能力，而不是在本项目内绕过基础库。

## Web 页面原则

- 所有页面必须兼容手机展示。
- 页面风格应简洁、大方、紧凑，优先方便实际操作。
- `/fan` 页面应重点服务高频操作：状态查看、速度设定、定时设定、取消定时、停止风扇。
- 速度设定使用“预设档位 + 数字输入”的方式。
- 定时设定使用“预设档位 + 数字输入”的方式。
- 按钮、输入框和表单控件必须视觉对齐，尤其是 `/fan` 与 `/config` 页面。
- 长内容状态需要拆分展示，例如 WiFi 信息中的 IP、RSSI，时间信息中的 Clock。
- 不要在页面中保留含义不清的按钮或文案；例如按钮名称必须能直接说明动作。
- 保存配置后必须有明确交互反馈，区分保存成功、无变化、保存失败。

## API 与路由约束

- Esp8266Base 当前应用 API 配额为 6 个，本项目注册 API 时必须注意配额。
- `/fan` 页面依赖 `/api/status` 轮询状态，该 API 必须注册。
- 不应为了保留低频 API 牺牲核心页面状态回显。
- 当前公开 API 包括：
  - `/api/status`
  - `/api/speed`
  - `/api/timer`
  - `/api/stop`
  - `/api/config`
  - `/api/ir/learn`

## 硬件与测试注意事项

- 用户可能在未接风扇硬件的情况下测试页面和 API。
- 未接风扇时，设置非零速度后可能因为没有 TACH/RPM 反馈触发堵转保护，这是正常保护行为。
- 调试速度、定时和回显问题时，应区分“页面/API 问题”和“未接硬件导致的堵转保护”。
- 红外模块在接入红外接收头前，应先做代码级检查；接入硬件后再做实际学习和按键测试。

## 低功耗原则

- 当前休眠使用 Esp8266Base 的 Modem Sleep 能力，保持 WiFi 和 Web 服务可用。
- 用户会用功率计观察功耗变化，因此低功耗相关结论要谨慎区分软件进入 sleep 状态和实际整机功耗变化。
- 如果功耗没有明显变化，应优先分析硬件供电、WiFi 模式和 Esp8266Base sleep 实现，不要在本项目里随意打补丁。
