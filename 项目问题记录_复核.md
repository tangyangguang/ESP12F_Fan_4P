## 四、本轮修复引入的轻微回归

### P0-03 修复在「堵转恢复成功」路径下会多算 `_run_duration`

P0-03 把 `_handleRunning` 的秒级计数改成「追赶式 while 循环」之后，所有「`_state` 变成 `SYS_RUNNING` 但 `_last_run_tick` 没被更新」的入口都会触发追赶，把这段时间统统算进 `_run_duration`。

唯一受影响的入口是堵转恢复成功路径：

[src/fan/FanController.cpp:376-378](src/fan/FanController.cpp:376)
```cpp
if (_fan.getRpm() > 0) {
    _state = SYS_RUNNING;   // 没有重置 _last_run_tick
    ESP8266BASE_LOG_I(...);
}
```

时序例子：
1. `t=0`，`setSpeed(50)`，`_applySpeed()` 把 `_last_run_tick=0`、`_state=RUNNING`。
2. `t=2600`，堵转触发：`_state=ERROR`，PWM 切到 0。
3. `t=3500`，用户重发 `setSpeed(50)` → 进入 ERROR 分支：`_recovery_attempting=true`、`_recovery_start_tick=3500`、`_fan.setSpeed(50)`，**但 `_state` 仍然是 `ERROR`**，且没有更新 `_last_run_tick`（仍然是 0 或上一次 _applySpeed 的值）。
4. `t=5000`，`_handleError` 检测到 RPM>0 → `_state=RUNNING`。
5. 同/下一 tick 进入 `_handleRunning`，`now - _last_run_tick` 跨度可能是 5 秒。while 循环把这 5 秒全部算进 `_run_duration`，但其中只有 1.5 秒（recovery 窗口）风扇真的在转，前面 ~2.5 秒是 ERROR 期 PWM=0。

旧代码漏秒 + 不追赶，刚好把这种「跨态」时间漏掉，所以问题不可见；新代码追赶逻辑暴露了原本就缺失的「进入 RUNNING 时重置计时」。

**影响**：`_run_duration` 在每次成功恢复堵转后会多记录几秒。F-20「运行时长统计」是 Should 需求且只精度到「至少累计」，不会让设备失能；但与「实际运行时长」的偏差会随着每次恢复事件慢慢累积。

**建议方向**（不在本次复核范围内修，仅提示）：在 `_handleError` 把 `_state` 改成 `SYS_RUNNING` 的同时，加一句 `_last_run_tick = millis();`；同样的处理也可以放进 `_applySpeed()` 里所有「state 变化为 RUNNING」的分支，作为不变量。

> 注：这个回归没有被现有测试捕获到。`test_controller_block_recovery_success_returns_to_running` 只断言 `_state == SYS_RUNNING`，没有断言 `_run_duration`。如果决定修，可以在该测试里加一条 `TEST_ASSERT_EQUAL(0, ctrl.getTotalRunDuration())` 或具体期望值。
