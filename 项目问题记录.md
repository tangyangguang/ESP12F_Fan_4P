## 一、状态机与行为逻辑

### P0-01 从 SYS_SLEEP 通过按键/红外被唤醒后，状态会被错误覆盖回 SYS_IDLE

**位置**：[src/fan/FanController.cpp:357-368](src/fan/FanController.cpp:357)

`_handleSleep()` 的实现是：

```cpp
void FanController::_handleSleep() {
    _processButtonEvents();
    _processIREvents();
    _processTimer();

    if (millis() - _last_operation_tick < 1000) {
        _is_sleeping = false;
        Esp8266BaseSleep::wakeModem();
        _state = SYS_IDLE;
        ESP8266BASE_LOG_I("FanCtrl", "Woken from sleep, transitioning to IDLE");
    }
}
```

按键短按时：
1. `_processButtonEvents()` → `setSpeed(speed)`；`setSpeed()` 内会调用 `_applySpeed()`，把 `_state` 改为 `SYS_RUNNING`。
2. 紧接着 `_handleSleep` 末尾的 `if (millis() - _last_operation_tick < 1000)` 成立（`setSpeed()` 刚刚把 `_last_operation_tick` 改为当前 `millis()`），把 `_state` 强制回写成 `SYS_IDLE`。

结果：从休眠态被本地按键 / 红外唤醒后，风扇 PWM 已经在跑，但 `getState()` 返回 `SYS_IDLE`，`/fan` 页面会显示「Idle」，`_handleRunning()` 的 `_run_duration` 累计也不会发生（直到下一次 setSpeed 让状态再次变更）。

测试 [test/test_fan_controller.cpp:860-880](test/test_fan_controller.cpp:860) `test_controller_wake_from_sleep` 没复现该路径——它从测试代码外部直接调用 `ctrl.setSpeed(25)` 后再调用 `ctrl.tick()`，此时 `_state` 已经不是 `SYS_SLEEP`，进入的是 `_handleRunning`，绕过了 bug。

**建议方向**：先判断「事件来源是否带来状态切换」，再决定是否覆盖 `_state`；或者把唤醒判断放在 `_processButtonEvents/_processIREvents` 之前。

### P0-02 `FanDriver` 软启动渐变最低速度被硬编码为 10%，不随 `fan_min_speed` 变化

**位置**：[src/fan/FanDriver.cpp:60-77](src/fan/FanDriver.cpp:60)

```cpp
if (gradient_speed < 10 && _target_speed >= 10) {
    gradient_speed = 10;  // Minimum effective speed
}
```

`fan_min_speed` 可被 `/config` 配置为 0-50%。如果用户把 `fan_min_speed` 设为 30，理论上软启动应该从 30% 起步，但代码仍然以 10% 起步，会经过一段「低于最低有效速度」的区间。

同时，`FanDriver` 自身的「最低有效速度」与 `FanController` 的 `_min_effective_speed` 是两个独立概念，`FanDriver` 没有 setter / getter 让 Controller 把值传进来，等于配置项与底层逻辑实际脱钩。

`FanDriver::tick()` 里的同一个 `10` 也用在 RUNNING/SOFT_STOP 之外的地方——总体上是把「最低有效转速」语义硬编进了 HAL，而 DOC-01 F-01 / F-15 明确这是可配置参数。

### P0-03 `_processTimer` / `_handleRunning` 的秒级计数在 tick 滞后时会丢秒

**位置**：[src/fan/FanController.cpp:333-355](src/fan/FanController.cpp:333) 与 [src/fan/FanController.cpp:477-494](src/fan/FanController.cpp:477)

```cpp
if (now - _last_run_tick >= 1000) {
    _run_duration++;
    _last_run_tick = now;
    ...
}
```

```cpp
if (now - _last_timer_tick >= 1000) {
    _timer_remaining--;
    _last_timer_tick = now;
    ...
}
```

这是「采样式」实现，而不是「追赶式」实现：如果某次 `tick()` 之间间隔 2.5s（比如刚做完一次 Flash 写、配置保存、OTA 期间），只会 `--` 一次，丢掉 1.5s。

DOC-01 F-06 要求「定时精度 ≤1min/24h」（≈0.07%），单次 Flash 写入 100ms 量级偶发不一定会突破指标；但当出现 OTA、堵转检测、`Esp8266Base` 内部慢操作时容易超标。

更严重的是 `_run_duration`——它是累积统计值，长时间运行后会持续偏小。

**建议方向**：把判断改成 `while (now - _last_xxx_tick >= 1000) { ... ; _last_xxx_tick += 1000; }`，让漏掉的秒数被累计补上。

### P0-04 Web 配置保存非原子，前面的字段已写入 Flash 后任一字段非法即返回 400

**位置**：[src/web/FanWeb.cpp:371-507](src/web/FanWeb.cpp:371)

`handleApiConfig()` 对每个参数依次执行：

```cpp
if (server.hasArg("min_speed")) {
    if (!parseUintArg(...)) {
        server.send(400, ..., "{\"ok\":false,\"error\":\"invalid min_speed\"}");
        return;
    }
    if (v != old) {
        _controller->setMinEffectiveSpeed(v);  // 立即 setInt -> Flash
        changed++;
    }
}
if (server.hasArg("soft_start")) { ... 同上 ... }
...
```

如果用户提交 `min_speed=15&soft_start=99999`，代码会先把 `min_speed` 写进 Flash（`setMinEffectiveSpeed` 内部走的是 `Esp8266BaseConfig::setInt`，非 deferred），然后在校验 `soft_start` 时返回 400。

前端会显示「Save failed」，但 `min_speed` 已经被改写了；用户不会知道部分字段已经持久化。

AGENTS.md 提到 Esp8266Base 的写入是「逐个 key 原子写入」，但**整个事务的语义（要么全成功要么全失败）在本项目层是缺失的**。建议在解析阶段先把所有字段一次性 parse + clamp 到栈上，校验全部通过后再统一应用。

### P0-05 LED 闪烁频率与 README/DOC-02 文档描述差一倍

**位置**：[src/fan/LedIndicator.h:43-44](src/fan/LedIndicator.h:43) 与 [src/fan/LedIndicator.cpp:96-110](src/fan/LedIndicator.cpp:96)

```cpp
static const uint32_t SLOW_BLINK_INTERVAL = 1000;  // 1Hz
static const uint32_t FAST_BLINK_INTERVAL = 200;   // 5Hz
```

`update()` 是「每经过 INTERVAL，翻转一次 `_blink_state`」。所以：

- 慢闪：每 1000ms 翻转 → 周期 2000ms → **0.5Hz**，不是 1Hz
- 快闪：每 200ms 翻转 → 周期 400ms → **2.5Hz**，不是 5Hz

DOC-01 F-23、DOC-02 §4.4、DOC-03 §2、README §LED 指示灯都写「故障快闪 5Hz / WiFi 慢闪 1Hz」。要么改实现（INTERVAL 减半），要么改文档。注释和频率定义也应同步。

测试 [test/test_fan_controller.cpp:472-495](test/test_fan_controller.cpp:472) 验证的是 1000ms / 200ms 翻转间隔——它锁定的是当前实现，与文档不符。

### P0-06 `/api/status` JSON 字段名与 DOC-04 文档定义全面不一致

**位置**：[src/web/FanWeb.cpp:294-310](src/web/FanWeb.cpp:294) vs [docs/DOC-04_代码设计说明.md:326](docs/DOC-04_代码设计说明.md:326)

DOC-04 §5 给出的示例：

```json
{"ok":true,"data":{"state":"running","gear":2,"speed":50,"timer_remaining":1800,"uptime":3600,"wifi_connected":true,"wifi_rssi":-65,"ip":"192.168.1.100","ntp_synced":true,"error_code":0}}
```

实际响应：

```json
{"ok":true,"data":{"state":"Running","speed":50,"target_speed":50,"timer_remaining":1800,"run_duration":N,"blocked":false,"ip":"...","rssi":-65,"clock":"2026-05-09 12:00:00","ir_learning":false,"ir_key":0,"ir_remaining":0,"ir_learn_seq":N,"ir_last_protocol":0,"ir_last_code":"0x..."}}
```

差异点：

1. `state` 实际是 PascalCase（"Running"），文档是 lowercase（"running"）。前端 JS 直接 `e('st',d.state)` 显示原值。
2. 文档承诺有 `gear`，实际没有；前端也没用，只是约定丢失。

同样 `/api/config` GET 文档说返回 `soft_start_time`、`block_detect_time`，实际是 `soft_start`、`block_detect`（[FanWeb.cpp:478](src/web/FanWeb.cpp:478) vs [DOC-04 §5](docs/DOC-04_代码设计说明.md)）。

**建议方向**：把 DOC-04 的 API 章节按代码现状重写，并把版本号 / 日期对齐；或者反过来调整代码字段名以匹配文档。

## 二、健壮性与边界条件

### P1-01 `parseIRCodeEntry` 解析 protocol 时未限制最大值即可写入，再被强转为 uint8_t

**位置**：[src/fan/FanController.cpp:40-54](src/fan/FanController.cpp:40)

```cpp
unsigned long parsedProtocol = strtoul(value, &end, 10);
if (end == value || *end != ':' || parsedProtocol > 255) return false;
```

虽然有 255 上限校验，但同函数 `setKeyCode` 把 protocol 存为 `uint8_t`。如果配置文件被外部篡改写入 `protocol=300:HEX`，会被拒绝；但写入 `protocol=255:HEX` 时 IRremoteESP8266 的 `decode_type_t` 是否包含 255 没有任何保护。校验只做了类型边界，没有做语义边界。

风险等级低（只有手动改配置文件能触发），但该字段也没有 fallback——读取失败的 key 不会清空内存里残留的旧值（因为 `_loadConfig` 里只在解析成功时调 `setKeyCode`）。

### P1-02 IR 学习成功保存时与 `_saveIRCode` 的 unchanged 判断口径不一致

**位置**：[src/fan/FanController.cpp:632-650](src/fan/FanController.cpp:632)

`_saveIRCode` 用 `snprintf("%u:%016llX", ...)` 生成新值，再用 `getStr` 读出旧值并 `strcmp` 比较，相同则跳过写入。但 `_loadConfig` 里读取出来的字符串可能没有左零（早先版本 / 手动写过的情况）。比如旧值 `0:E01F`、新值 `0:000000000000E01F`，二者代表同一编码却 `strcmp` 不相等，会被识别为「变化」并触发一次 Flash 写。

学习极少做，影响很小，但属于「比较器和写入器格式没同步约束」的潜在 bug，建议在比较前做一次格式标准化。

### P1-03 `FanDriver::_tach_count` 在 ISR 与主循环之间是非原子读改清

**位置**：[src/fan/FanDriver.cpp:42-58](src/fan/FanDriver.cpp:42) 与 [src/fan/FanDriver.cpp:192-196](src/fan/FanDriver.cpp:192)

```cpp
if (elapsed > 0 && _tach_count > 0) {
    _rpm = static_cast<uint16_t>((_tach_count * 30000UL) / elapsed);
} else {
    _rpm = 0;
}
_tach_count = 0;
```

主循环在 `tick()` 内读取 `_tach_count`，计算 `_rpm`，然后清零。如果 ISR 在「读」与「清零」之间触发，会增加一个新值，但被随后清零吞掉。500ms 周期下漏掉 1-2 个脉冲不影响 RPM 一阶精度，但会出现毛刺，且 `volatile` 不保证原子性。

ESP8266 的简易处理是 `noInterrupts() { local = _tach_count; _tach_count = 0; } interrupts();` 配合 `IRAM_ATTR ISR`。

### P1-04 `setSpeed()` 不检查 speed 上限就写入 `_target_speed`

**位置**：[src/fan/FanController.cpp:184-212](src/fan/FanController.cpp:184)

```cpp
bool FanController::setSpeed(uint8_t speed) {
    ...

```

没有 `if (speed > 100) speed = 100`。`FanDriver::setSpeed` 自己会 clamp 到 100 写入 PWM，但 `_target_speed` 仍会保留 200。`/api/status` 里 `target_speed=200` 会让前端显示和数字输入框出现非法值。

### P1-05 BOOT 键缺少 `pinMode(BOOT_PIN, INPUT_PULLUP)` 和软件防抖

**位置**：[src/main.cpp:88-100](src/main.cpp:88)

```cpp
bool currentState = digitalRead(BOOT_PIN);
if (currentState == LOW && !isPressed) {
    pressStartTime = millis();
    isPressed = true;
} else if (currentState == HIGH && isPressed) {
    isPressed = false;
} else if (currentState == LOW && isPressed && (millis() - pressStartTime >= PRESS_DURATION_MS)) {
    ...
    delay(500);
    ESP.restart();
}
```

问题：

1. 没有显式 `pinMode(BOOT_PIN, INPUT_PULLUP)` 或 `INPUT`。GPIO0 上电后默认状态依赖外部下拉/上拉，硬件正常时是高，但软件没显式声明，可读性差。
2. 没有防抖。手指松开瞬间的抖动可能在 1s 阈值附近多次进出 LOW/HIGH，这里靠 `isPressed` 的边沿检测吃掉一部分，但对发生在「将到 1s」时刻的抖动不可靠。
3. `delay(500)` 是阻塞调用，会让 Esp8266Base 的 watchdog/web 短时间不可服务。一般 WDT 足够长能熬过去，但仍是可优化点。

### P1-06 IRReceiverDriver 把 IRrecv 放在静态函数局部变量里延迟初始化

**位置**：[src/fan/IRReceiverDriver.cpp:42-49](src/fan/IRReceiverDriver.cpp:42)

```cpp
static IRrecv irrecv(_recv_pin, kCaptureBufferSize, kCaptureTimeout, true);
static bool initialized = false;
if (!initialized) {
    irrecv.enableIRIn();
    initialized = true;
    ...
}
```

问题：

1. `_recv_pin` 是成员变量。函数级 `static` 只在第一次进入时构造一次；如果将来出现第二个 `IRReceiverDriver` 实例，第二个实例的 pin 会被无声忽略，IR 仍接到第一个实例对应的 GPIO。当前项目只构造一次，没暴露 bug，但属于设计陷阱。
2. `enableIRIn()` 应该在 `begin()` 阶段做，而不是在循环中懒加载——README、DOC-03 都说「`begin()` 完成硬件初始化」，这违反了模块约定。
3. `kCaptureBufferSize`、`kCaptureTimeout` 写在头文件作用域 `static const`，每个翻译单元都会有一份独立副本，不大但属于轻微浪费。

### P1-07 `setSpeed()` 在 SYS_ERROR 状态下与 `attemptBlockRecovery()` 内部代码重复

### P1-10 ButtonDriver 长按 5s 之后单独释放某个按键时按键事件不会再触发

**位置**：[src/fan/ButtonDriver.cpp:78-108](src/fan/ButtonDriver.cpp:78)

`_both_long_triggered` 仅在 **两个按键都释放** 时清零：

```cpp
if (!_accel_pressed && !_decel_pressed) {
    _both_long_triggered = false;
}
```

而短按检查里要求 `!_both_long_triggered`：

```cpp
if (duration < 1000 && !_both_long_triggered) {
    return BTN_ACCEL_SHORT;
}
```

序列：

1. 同时按下 A+B，超过 5s 触发 BTN_BOTH_LONG，`_both_long_triggered = true`。
2. 用户先松开 A（短时间内），再松开 B；
3. **第二步松开 B 之后**，立刻按下 A（短按）；
4. A 的「按下→松开」会经历 `_accel_pressed = true → false`，但因为 B 还没松开过——其实步骤 2 中 B 已松开，所以这里不再是 bug，而是步骤 2 中**释放 A 时**，由于 `_both_long_triggered` 仍为 true，A 的释放被吃掉、不发短按事件。这一点符合预期（防误触）。

真正的小问题是：**`_decel_pressed` 之类的内部布尔可能因为「按键还没松开就 trigger 了 long press」而仍为 true**，而清除条件又只在「都松开」时生效——如果用户长按之后立刻松开 B 但 A 仍按住一直不放，`_accel_pressed` 持续为 true，`_both_long_triggered` 也持续为 true，再用 B 短按时其释放事件因为 `_both_long_triggered` 仍 true 而被吞掉。

实际人机交互极少落入此场景，标记为 P2 也可，这里给 P1 是因为代码里没注释说明这是「故意吃掉」还是「漏掉」。

### P2-08 `handleApiStatus` 用 `%08llX` 输出 64 位 IR 编码，最少 8 位但常见编码可能更长

**位置**：[src/web/FanWeb.cpp:299](src/web/FanWeb.cpp:299) 与 [src/web/FanWeb.cpp:553](src/web/FanWeb.cpp:553)

### P2-10 `Esp8266BaseConfig::clearAll()` 在 factory reset 时会同时清掉 WiFi 凭证

**位置**：[src/fan/FanController.cpp:251-257](src/fan/FanController.cpp:251)

```cpp
bool FanController::resetFactory() {
    if (!Esp8266BaseConfig::clearAll()) {
        ESP8266BASE_LOG_E("FanCtrl", "Factory reset failed: config clearAll failed");
    }
    ESP.restart();
    return true;
}
```

`clearAll()` 是 Esp8266Base 提供的「清空所有 KV」，会一并清掉 `eb_wifi_*`、`eb_web_pass`、`fan_*` 等。从「恢复出厂设置」语义上没毛病，但用户按了双键 5s 后 WiFi、Web 密码全部丢失，比想象的更激进。文档（DOC-01 F-04）写「恢复出厂设置」但没有显式列出会清掉哪些类。

如果希望保留 WiFi 凭证只清风扇配置，需要在 Esp8266Base 提供一个「清前缀」的 API；按 AGENTS.md 「不得修改 Esp8266Base」的原则，应在 Esp8266Base 那里提需求。

### P2-12 `handleConfigPage` 把 `"selected"` / `""` 用 `sendContent_P` 发出去

**位置**：[src/web/FanWeb.cpp:262-265](src/web/FanWeb.cpp:262)

```cpp
Esp8266BaseWeb::sendContent_P(_controller->getAutoRestore() ? "selected" : "");
Esp8266BaseWeb::sendContent_P(CONFIG_AUTO_END);
Esp8266BaseWeb::sendContent_P(_controller->getAutoRestore() ? "" : "selected");
Esp8266BaseWeb::sendContent_P(CONFIG_AUTO_END2);
```

`sendContent_P` 期望 PROGMEM 地址。在 ESP8266 现代 Arduino core 上字符串字面量本身一般落在 irom flash，能通过 `pgm_read_byte` 读出，所以**实际多半工作正常**；但接口语义并不保证字符串字面量都是 `PROGMEM`。更安全的做法是 `sendChunk(...)`（普通字符串）或显式 `PROGMEM` 化常量。

## 九、复测建议

完成本轮评估只用静态阅读，未运行测试与硬件。建议下一步动作（顺序）：

1. 直接跑 `pio test -e native`，确认 P0-01、P0-03、P0-05 三项以测试反推；如果测试通过，说明现有断言已锁定错误实现，需要先修测试。
2. `pio run -e esp12f` 确认与新版 Esp8266Base 编译没退化（特别是 `lib_deps = symlink://../Esp8266Base` 在 CI 环境会失败，本地 OK）。
3. 在带硬件场景下复现：
   - 触发 SLEEP 后短按按键，看 `/fan` 页面 state 是否短暂是 Idle 然后被 RUNNING 回写、还是一直停留在 Idle（验证 P0-01）。
   - `/config` 提交一个非法 `soft_start=99999`，提交前用 GET `/api/config` 记下 `min_speed`，提交后再 GET，看 `min_speed` 是否被部分写入（验证 P0-04）。
   - 用相机或频率计观察 LED 实际闪烁频率（验证 P0-05）。
